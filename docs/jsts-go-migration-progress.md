## How To Read The Table

- `Sonar`: Sonar rule key.
- `Impl`: Rule implementation shape. `original` means native SonarJS, `external` is a direct upstream rule, `decorated` wraps an upstream rule with SonarJS logic, and `composite` combines multiple upstream rules.
- `Source`: `original` for native SonarJS rules, otherwise the upstream plugin or plugins wrapped by SonarJS.
- `Upstream`: Direct upstream rule id or ids used by the SonarJS rule.
- `Gen typed`: Whether `generated-meta.ts` sets `meta.docs.requiresTypeChecking`.
- `Config`: Whether the rule has a local `config.ts`.
- `Defaults`: Whether `generated-meta.ts` declares non-empty `defaultOptions`.
- `Migrated`: Whether the rule is already routed through `JSTS_GO_RULES` in `JsTsChecks.java`.
- `Type svc`: Aggregate type-service signal. `Yes` means the rule is marked typed in `generated-meta.ts` or directly uses parser services / the TypeScript checker in the local SonarJS implementation or direct upstream implementation.
- `Ext AST`: Aggregate extended-AST signal. `Yes` means the local SonarJS implementation or direct upstream implementation uses non-plain-ESTree shapes, including `TSESTree`, TS nodes, and local JSX-specific checks. Plain JSX-only third-party rules are treated as safe to stay in ESLint, so this signal only counts TS-specific or parser-specific shapes on upstream rules.
- `Subject to migration`: Conservative migration signal. `Yes` means `Type svc = Yes`, `Possibly` means `Type svc = No` and `Ext AST = Yes`, and `No` means neither signal is present.
- `Notes`: Composite wrappers, bundled upstream scans, local-wrapper-added dependencies, or mismatches between generated metadata and direct implementation usage. When the local/upstream origin still matters, it is called out here.

## Migration Decision Notes

- `Subject to migration` now follows the conservative implementation view: `Yes` means a hard type-service dependency, while `Possibly` means an AST/parser dependency without type services.
- A rule with `Subject to migration = Possibly` can stay in Node if Node keeps a sufficiently compatible TS parser/runtime.
- If a rule's `generated-meta.ts` `languages` contains `ts`, that rule still needs some TS-capable runtime once the Node analyzer stops parsing TS, even when `Subject to migration` is `No`.
- In other words, a rule can be safe to stay in ESLint from this implementation-dependency perspective and still need a TS execution path for coverage reasons.
- Additional factors that can still push a rule toward `jsts-go` even when both `Type svc` and `Ext AST` are `No` include runtime dependence on the `typescript-eslint` or `typescript` parser stack, TS-specific scope semantics not captured by the current heuristics, composite/decorator coupling that is easier to preserve on the Go side, and fixer parity that depends on TS parser behavior.
- If SonarJS later adopts another JS-side TS parser, some syntax-only TS rules could still stay in Node. That would not change the need to migrate rules that depend on type services or current TS parser semantics.

## Parser Compatibility Caveat

- For AST-only rules, "can stay in Node" does not just mean "the replacement parser accepts TypeScript syntax".
- Babel's native parser output is a Babel AST with documented deviations from ESTree; Babel provides an ESTree-oriented adapter layer for ESLint use-cases instead of promising that its native AST already matches the ESTree ecosystem contract.
- `typescript-eslint` similarly does more than just parse TypeScript syntax: `typescript-estree` converts the TypeScript AST into an ESTree-compatible form, and the `typescript-eslint` ESLint plugin expects `@typescript-eslint/parser` to provide those ESLint-compatible nodes and, when needed, backing services.
- ESLint custom parsers may also need to provide `visitorKeys`, `scopeManager`, and parser `services`, so compatibility is about traversal and scope behavior in addition to node shapes.
- Therefore, AST-only `typescript-eslint` rules should be read as `Possibly`, not automatically safe: they can remain in Node only if the replacement parser/runtime reproduces enough of the current TSESTree/ESLint parser contract for the specific rules we keep.

## Practical Planning Buckets

- `Must migrate`: any rule with `Type svc = Yes`.
- `Can stay in Node only if Node keeps TS parser support`: any rule with `Type svc = No` and `Ext AST = Yes`. That includes untyped `typescript-eslint` rules such as `S2094` when the only dependency is TS parser/AST compatibility.
- `Safe in JS-only Node`: any rule with `Type svc = No` and `Ext AST = No`, subject to the extra caveats listed above about parser/runtime compatibility.
- This planning split now matches the `Subject to migration` column: hard migration is driven by type-service dependency, while AST-only rules remain conditional on TS parser/runtime support in Node.
